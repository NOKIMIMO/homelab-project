# Homelab — Functional Specification

## 1. Snapshot
Homelab is a self‑hosted, no-code / low-code platform to assemble independent modules. It provides:
- a web front to design and run modules,
- a core runtime and SDK exposing auth, storage, DB and proxy services,
- an extensible plugin system that adds reusable actions and UI building blocks.

Each user runs and manages their own Homelab instance and installs only the modules/plugins they want.

---

## 2. Vision & Goals
- Enable rapid module creation through configuration and function composition (no custom core code required).
- Provide secure, per‑instance extensibility via sandboxed plugins.
- Make deployment and operation simple: each instance is autonomous.
- Offer a stable runtime + API so third‑party plugins can extend functionality safely.

---

## 3. Core Concepts
- **Module**: Discovery unit described by a manifest (e.g. `manifest.json`) and optional UI/data files (example: `photos.xml`).
- **Function / Action**: Reusable operations (built‑ins: `UPLOAD_FILE`, `GET_FILE`, `CREATE`, `READ`, `LIST`, `UPDATE`, `DELETE`); modules compose logic steps referencing action types.
- **Plugin**: Package registering new actions, validators and UI widgets for that instance.
- **Data object**: Table definition (`TableDefinition`, `ColumnDefinition`, `AutoGeneratedColumn`) used by `GenericTableLayer`.
- **Runtime / SDK**: Core services (auth, storage, module DB, telemetry, proxy) accessible by modules/plugins.

---

## 4. High-level Architecture
- **Backend**: Kotlin + Spring Boot — module scanning, lifecycle, proxy, auth (JWT + challenge/signature), file handling, telemetry.
- **Frontend**: Web UI (React/TypeScript recommended) for module design, plugin management and orchestration.
- **Plugin Registry**: local to each instance; discover/install plugins, expose metadata (id, version, actions).
- **Module SDK**: client libraries so modules/plugins call core services safely.

---

## 5. Module & Function Model
- Manifest keys (examples): `id`, `name`, `actions`, `dataObjects`, `permissions`.
- Function declaration (`ModuleActionDeclaration`): `name`, `description`, `parameters`, `logic`, `actUponObject`.
- Logic step (`ModuleActionLogic`): `type` (maps to an action implementation) and `parameters` (bindings or literal values).
- Controller resolves logic parameter bindings from request before invoking `AppletService`.

---

## 6. Built‑in Actions (examples)
- `UPLOAD_FILE` — handle multipart upload, store file to `data/{moduleId}` and create data row.
- `GET_FILE` — retrieve file metadata and return a downloadable file.
- `LIST`, `READ`, `CREATE`, `UPDATE`, `DELETE` — operate on `GenericTableLayer` rows.

Files of interest: `UploadFileAction.kt`, `GetFileAction.kt`, `ListAction.kt`, `ReadAction.kt`, `DeleteAction.kt`.

---

## 7. Plugin Model
- Package contains metadata (id, version, actions), implementation entrypoint and optional validators/UI widgets.
- Installation via UI / API (`/api/plugins/*`).
- Plugins are sandboxed: no direct FS/DB access — must use core storage/DB SDK and declare required capabilities.

---

## 8. API Surface (overview)
- **Authentication**
  - `/api/auth/*` — login (password or challenge/signature), signup requests, user management.
- **Module management**
  - `/api/modules/*` — scan, list, install, start/stop, health.
- **Function invocation**
  - `/api/{module}/{function}` — JSON or multipart invocation handled by `AppletControler`.
- **Proxy**
  - `/api/proxy/{module}/**` — forward requests to module internal URL (token validated).
- **Telemetry**
  - `/api/telemetry` — system metrics.
- **Plugin management**
  - `/api/plugins/*` (to be implemented).

Key controller files: `AppletControler.kt`, `ModuleController.kt`, `ProxyController.kt`, `AuthController.kt`, `TelemetryController.kt`.

---

## 9. Data & Storage
- `GenericTableLayer` abstracts persistence:
  - Uses `ModuleDatabaseService` when available, or in‑memory callbacks otherwise.
  - `TableDefinition` drives validation: columns, nullable, regex and `AutoGeneratedColumn` types (`ID`, `CREATED_AT`, `UPDATED_AT`, `FILE`).
- File handling: uploads saved under `data/{moduleId}`; rows store file path metadata. All file operations should be centralized via a `StorageService`/SDK.

Relevant classes: `GenericTableLayer.kt`, `TableDefinition.kt`, `ColumnDefinition.kt`, `AutoGeneratedColumn.kt`.

---

## 10. Security & Permissions
- Authentication: JWT tokens (header, cookie, or query param). Challenge/signature login supported.
- Authorization: module‑level permissions declared in module manifest (e.g. `read:photos`).
- Plugin least‑privilege model: plugins declare capabilities; runtime enforces them.
- Proxy only forwards to registered internal module URLs after token validation (`JwtAuthenticationFilter.kt`, `SecurityConfig.kt`).

---

## 11. UX / Key Use Cases
- **No‑code module creation**
  - Via UI select data schema, choose available actions/plugins, bind logic steps and map parameters.
  - Save → core scans and registers module.
- **Typical flows**
  - Upload photo: choose `UPLOAD_FILE` action, map `file` to target field → core stores file and creates row.
  - Download photo: call function composed with `GET_FILE` → core returns file stream.
- **Plugin install**: plugin adds action `image-resize` — UI shows it in composer for new functions.

Module example manifest: `module-example/manifest.json`, UI/data: `module-example/photos.xml`.

---

## 12. Telemetry & Observability
- Collect module health, resource metrics (CPU/RAM/disk), plugin events and logs.
- Expose per‑module logs and health check results; circuit breaker for unstable modules.

---

## 13. Non‑functional Requirements
- Extensibility: versioned stable plugin API.
- Security: least‑privilege plugin model, JWT + cookie support.
- Resilience: timeouts, graceful failures, circuit breakers for module proxying.
- Usability: visual function composer, automatic form generation from declarations.

---

## 14. Mapping to Codebase / Implementation Notes
- Keep `ModuleActionDeclaration` and `ModuleActionLogic` as canonical contracts (see `ModuleActionDeclaration.kt`, `ModuleActionLogic.kt`).
- Central action registry keyed by action `type`; built‑ins register on startup; plugins register additional factories.
- Centralize file access in a `StorageService` used by `UploadFileAction` and `GetFileAction`.
- Controller flow:
  - `AppletControler` resolves bindings → `AppletService.invokeFunctionOfModule` executes resolved logic using action implementations.
- Config entries: `HomelabConfig.kt` sets `modulesScanPath`; app properties in `application.properties`.
