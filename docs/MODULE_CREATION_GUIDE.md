# Guide de création d'un module

## Structure d'un module

Un module est un dossier placé dans le répertoire scanné par le backend (configurable via `HOMELAB_MODULES_SCAN_PATH`). Il contient au minimum :

```
modules/
└── mon-module/
    ├── manifest.json        ← obligatoire : déclare le module
    ├── ma_table.xml         ← schéma de base de données
    ├── mon_ui.json          ← interface utilisateur
    ├── params.json          ← paramètres configurables (optionnel)
    └── icone.svg            ← icône affichée dans le dashboard (optionnel)
```

Le backend détecte et charge automatiquement le module au démarrage. Aucune recompilation n'est nécessaire.

---

## 1. `manifest.json` — Déclaration du module

```json
{
    "id": "mon-module",
    "name": "Mon Module",
    "version": "1.0.0",
    "icon": "icone.svg",
    "description": "Description courte affichée dans le dashboard.",

    "actions": [
        {
            "visual": "ma_table.xml",
            "functions": [
                {
                    "name": "creerEntree",
                    "description": "Créer une nouvelle entrée.",
                    "parameters": [],
                    "logic": [{ "type": "CREATE" }],
                    "actUponObject": "ma_table.xml"
                },
                {
                    "name": "listerEntrees",
                    "description": "Lister toutes les entrées.",
                    "parameters": [],
                    "logic": [{ "type": "LIST" }],
                    "actUponObject": "ma_table.xml"
                }
            ]
        }
    ],

    "page": "mon_ui.json",
    "dataObjects": ["ma_table.xml"],
    "permissions": [
        "read:mon-module",
        "write:mon-module",
        "delete:mon-module"
    ]
}
```

### Champs

| Champ | Obligatoire | Description |
|---|---|---|
| `id` | oui | Identifiant unique (snake-case ou kebab-case, sans espaces) |
| `name` | oui | Nom affiché dans le dashboard |
| `version` | non | Semver libre |
| `icon` | non | Fichier SVG ou image dans le dossier du module |
| `description` | non | Texte court affiché dans le dashboard |
| `actions` | oui | Tableau des groupes de fonctions |
| `page` | oui | Fichier JSON de l'interface |
| `dataObjects` | oui | Liste des fichiers XML de schémas utilisés |
| `permissions` | non | Permissions déclaratives (non encore enforced) |

### Paramètre d'une fonction (`parameters[]`)

```json
{
    "name": "id",
    "type": "EQUAL",
    "description": "ID de l'entrée",
    "optional": false
}
```

Le champ `type` indique comment le paramètre est utilisé **en base de données** :

| Type | Comportement |
|---|---|
| `EQUAL` | Filtre `WHERE col = valeur` |
| `NONE` | Passé à l'action mais non utilisé comme filtre SQL |

> **`actUponObject` est obligatoire** pour chaque fonction. Le backend échoue au démarrage si le fichier XML n'existe pas.

## 2. `ma_table.xml` — Schéma de base de données

Le backend génère automatiquement une table SQL à partir de ce fichier.

```xml
<ma_table>

    <!-- Colonnes auto-générées (toujours inclure) -->
    <autoGenerated>
        <id/>
        <createdAt/>
        <updatedAt/>
    </autoGenerated>

    <!-- Colonne simple -->
    <col>
        <naming val="titre"/>
        <typing val="string"/>
    </col>

    <!-- Colonne avec contraintes -->
    <col>
        <naming val="slug"/>
        <typing val="string"/>
        <unique val="true"/>
        <nullable val="false"/>
    </col>

    <!-- Colonne numérique -->
    <col>
        <naming val="score"/>
        <typing val="integer"/>
    </col>

</ma_table>
```

### Types de colonnes (`typing`)

| Valeur | Type SQL généré |
|---|---|
| `string` | VARCHAR / TEXT |
| `integer` | INTEGER |
| `boolean` | BOOLEAN |
| `float` | DOUBLE / FLOAT |

### Contraintes disponibles

| Balise | Valeur | Effet |
|---|---|---|
| `<unique>` | `"true"` | Contrainte UNIQUE sur la colonne |
| `<nullable>` | `"false"` | Contrainte NOT NULL |

### Colonnes auto-générées (`autoGenerated`)

| Balise | Contenu généré |
|---|---|
| `<id/>` | UUID auto-généré, clé primaire |
| `<createdAt/>` | Timestamp de création |
| `<updatedAt/>` | Timestamp de dernière mise à jour |
| `<file/>` | Stockage binaire (pour fichiers uploadés) |

## 3. `mon_ui.json` — Interface utilisateur

```json
{
  "page": {
    "id": "mon_module_page",
    "layout": "vertical",

    "bindings": {
      "maListe": "listerEntrees"
    },

    "state": {
      "champRecherche": ""
    },

    "components": [...]
  }
}
```

### `bindings`

Associe un nom local à une fonction du module. Ce nom est utilisé comme `source.binding` dans les composants.

```json
"bindings": {
  "maListe": "listerEntrees"
}
```

### `state`

Valeurs initiales des variables de l'interface. Elles peuvent être lues et écrites par les composants via `{{nomVariable}}`.

---

## 4. Composants UI disponibles

### `Header`
Titre de la page.

```json
{
  "type": "Header",
  "props": { "title": "Mon Module" }
}
```

---

### `ActionBar`
Barre d'actions horizontale. Contient d'autres composants.

```json
{
  "type": "ActionBar",
  "components": [...]
}
```

---

### `Button`
Bouton cliquable avec action optionnelle.

```json
{
  "type": "Button",
  "props": {
    "label": "Rafraîchir",
    "icon": "refresh"
  },
  "action": {
    "action": "listerEntrees",
    "method": "POST"
  }
}
```

**Props disponibles :**
- `label` — texte du bouton
- `icon` — nom d'icône Lucide (ex: `"trash"`, `"upload"`, `"refresh"`, `"plus"`)
- `uploadTarget` — ID d'une `FileUploadZone` à déclencher

---

### `TextInput`
Champ de texte avec bouton de soumission. Lié au state via `stateKey`.

```json
{
  "type": "TextInput",
  "props": {
    "placeholder": "Rechercher...",
    "stateKey": "champRecherche",
    "buttonLabel": "Go"
  },
  "action": {
    "action": "rechercherEntrees",
    "method": "POST",
    "params": { "query": "{{champRecherche}}" }
  }
}
```

**Props disponibles :**
- `stateKey` — **obligatoire** : clé dans `state` lue et écrite par ce champ
- `placeholder` — texte fantôme
- `label` — libellé au-dessus du champ
- `buttonLabel` — texte du bouton (défaut : `"Rechercher"`)

L'`action` se déclenche sur Enter ou clic du bouton.

---

### `List`
Liste d'éléments provenant d'un binding.

```json
{
  "type": "List",
  "props": { "emptyMessage": "Aucun élément." },
  "source": {
    "binding": "maListe",
    "method": "POST"
  },
  "item": {
    "type": "ListItem",
    "props": {
      "title": "{{nom}}",
      "subtitle": "{{description}}"
    },
    "actions": [
      {
        "type": "IconButton",
        "props": { "icon": "trash", "tooltip": "Supprimer" },
        "action": [
          { "action": "supprimerEntree", "method": "POST", "params": { "id": "{{id}}" } },
          { "action": "listerEntrees", "method": "POST" }
        ]
      }
    ]
  }
}
```

**`{{variable}}`** — interpolation depuis les données de la ligne courante. `{{id}}` est toujours disponible.

**`defaultClick`** — action déclenchée au clic sur la ligne :

```json
"defaultClick": {
  "action": "lireEntree",
  "method": "POST",
  "params": { "id": "{{id}}" },
  "then": {
    "setState": { "entreeSelectionnee": "{{result}}" }
  }
}
```

---

### `IconButton`
Bouton icône (utilisé dans les `actions` d'un `ListItem`).

```json
{
  "type": "IconButton",
  "props": { "icon": "trash", "tooltip": "Supprimer" },
  "action": { "action": "supprimerEntree", "method": "POST", "params": { "id": "{{id}}" } }
}
```

---

### `FileUploadZone`
Zone de dépôt de fichier. Associée à un bouton via `uploadTarget`.

```json
{
  "type": "FileUploadZone",
  "props": {
    "id": "mon-upload",
    "accept": "image/*",
    "multiple": false
  },
  "action": [
    { "action": "ajouterFichier", "method": "POST" },
    { "action": "listerEntrees", "method": "POST" }
  ]
}
```

---

### `Modal`
Fenêtre modale conditionnelle.

```json
{
  "type": "Modal",
  "visible": "{{entreeSelectionnee != null}}",
  "content": {
    "type": "ImageViewer",
    "source": { "binding": "getFichier", "method": "POST" },
    "params": { "id": "{{entreeSelectionneeId}}" }
  },
  "actions": [
    {
      "type": "Button",
      "props": { "label": "Fermer" },
      "action": { "type": "setState", "target": "entreeSelectionnee", "value": null }
    }
  ]
}
```

`visible` accepte une expression `{{...}}` évaluée sur le state.

---

### `ImageViewer`
Affichage d'image (depuis un binding de type `GET_FILE`).

```json
{
  "type": "ImageViewer",
  "source": { "binding": "getFichier", "method": "POST" },
  "params": { "id": "{{id}}" },
  "props": { "alt": "{{file_name}}" }
}
```

---

## 5. Types d'actions backend (`logic[].type`)

| Type | Comportement |
|---|---|
| `LIST` | Retourne toutes les lignes de la table |
| `READ` | Retourne une ligne filtrée par les paramètres `EQUAL` |
| `CREATE` | Insère une nouvelle ligne avec les paramètres fournis |
| `DELETE` | Supprime la/les lignes filtrées par les paramètres `EQUAL` |
| `UPLOAD_FILE` | Upload un fichier binaire et crée une entrée en base |
| `GET_FILE` | Retourne le fichier binaire d'une entrée |
| `FETCH_EXTERNAL` | Appelle une API HTTP externe, parse la réponse, upsert en base |

---

## 6. `params.json` — Paramètres configurables (optionnel)

Permet d'exposer des valeurs configurables depuis l'UI (icône ⚙️ sur la carte du dashboard).

```json
{
    "parameters": [
        {
            "key": "apiKey",
            "label": "Clé API",
            "type": "secret",
            "defaultValue": "",
            "description": "Clé secrète pour l'accès à l'API externe"
        },
        {
            "key": "baseUrl",
            "label": "URL de base",
            "type": "string",
            "defaultValue": "https://api.exemple.com/v1",
            "description": "Endpoint de l'API"
        }
    ]
}
```

### Types de paramètres

| Type | Comportement |
|---|---|
| `string` | Champ texte standard, valeur visible |
| `secret` | Champ texte masqué, valeur cachée dans l'UI |

### Lire un paramètre côté backend

Les paramètres sont accessibles dans une action via `GlobalParametersService` :

```kotlin
val apiKey = globalParametersService.getParam(moduleId, "apiKey")
    ?: return mapOf("error" to "Paramètre 'apiKey' non configuré")
```

---

## 7. Actions chaînées

Une `action` peut être un tableau pour exécuter plusieurs actions en séquence. Exemple : supprimer puis rafraîchir la liste.

```json
"action": [
  { "action": "supprimerEntree", "method": "POST", "params": { "id": "{{id}}" } },
  { "action": "listerEntrees", "method": "POST" }
]
```

## 8. Exemple complet — module "Tâches"

### Structure

```
modules/tasks/
├── manifest.json
├── tasks.xml
├── tasks_ui.json
```

### `manifest.json`

```json
{
    "id": "tasks",
    "name": "Tâches",
    "version": "1.0.0",
    "description": "Gestionnaire de tâches simple.",
    "actions": [
        {
            "visual": "tasks.xml",
            "functions": [
                {
                    "name": "createTask",
                    "description": "Créer une tâche.",
                    "parameters": [
                        { "name": "title", "type": "NONE" },
                        { "name": "done", "type": "NONE" }
                    ],
                    "logic": [{ "type": "CREATE" }],
                    "actUponObject": "tasks.xml"
                },
                {
                    "name": "listTasks",
                    "description": "Lister toutes les tâches.",
                    "parameters": [],
                    "logic": [{ "type": "LIST" }],
                    "actUponObject": "tasks.xml"
                },
                {
                    "name": "deleteTask",
                    "description": "Supprimer une tâche.",
                    "parameters": [{ "name": "id", "type": "EQUAL" }],
                    "logic": [{ "type": "DELETE" }],
                    "actUponObject": "tasks.xml"
                }
            ]
        }
    ],
    "page": "tasks_ui.json",
    "dataObjects": ["tasks.xml"],
    "permissions": ["read:tasks", "write:tasks", "delete:tasks"]
}
```

### `tasks.xml`

```xml
<tasks>
    <autoGenerated>
        <id/>
        <createdAt/>
        <updatedAt/>
    </autoGenerated>

    <col>
        <naming val="title"/>
        <typing val="string"/>
        <nullable val="false"/>
    </col>

    <col>
        <naming val="done"/>
        <typing val="boolean"/>
    </col>
</tasks>
```

### `tasks_ui.json`

```json
{
  "page": {
    "id": "tasks_page",
    "layout": "vertical",
    "bindings": { "taskList": "listTasks" },
    "state": { "newTaskTitle": "" },
    "components": [
      { "type": "Header", "props": { "title": "Tâches" } },
      {
        "type": "ActionBar",
        "components": [
          {
            "type": "TextInput",
            "props": {
              "placeholder": "Nouvelle tâche...",
              "stateKey": "newTaskTitle",
              "buttonLabel": "Ajouter"
            },
            "action": [
              {
                "action": "createTask",
                "method": "POST",
                "params": { "title": "{{newTaskTitle}}", "done": false }
              },
              { "action": "listTasks", "method": "POST" }
            ]
          }
        ]
      },
      {
        "type": "List",
        "props": { "emptyMessage": "Aucune tâche pour l'instant." },
        "source": { "binding": "taskList", "method": "POST" },
        "item": {
          "type": "ListItem",
          "props": { "title": "{{title}}" },
          "actions": [
            {
              "type": "IconButton",
              "props": { "icon": "trash", "tooltip": "Supprimer" },
              "action": [
                { "action": "deleteTask", "method": "POST", "params": { "id": "{{id}}" } },
                { "action": "listTasks", "method": "POST" }
              ]
            }
          ]
        }
      }
    ]
  }
}
```

## Checklist de création

- [ ] Dossier créé dans le répertoire des modules
- [ ] `manifest.json` — `id`, `name`, `actions`, `page`, `dataObjects` renseignés
- [ ] Chaque fonction a un `actUponObject` pointant vers un fichier XML existant
- [ ] Fichier(s) XML créés avec au moins `<autoGenerated><id/></autoGenerated>`
- [ ] Fichier UI JSON créé avec `page.id` et `page.components`
- [ ] Backend redémarré pour scanner le nouveau module
