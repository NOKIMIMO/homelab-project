Guide de création d'un module

Tout ce qu'il faut savoir pour créer un nouveau module sans toucher au backend.

Structure d'un module

Un module est un simple dossier placé dans le répertoire scanné par le backend (HOMELAB_MODULES_SCAN_PATH). Le backend le détecte et le charge automatiquement au démarrage — aucune recompilation nécessaire.

modules/
└── mon-module/
    ├── manifest.json      ← obligatoire : déclare le module
    ├── ma_table.xml      ← schéma de base de données
    ├── mon_ui.json       ← interface utilisateur
    ├── params.json       ← paramètres configurables (optionnel)
    └── icone.svg        ← icône affichée dans le dashboard (optionnel)

1 manifest.json — Déclaration du module

{
    "id":          "mon-module",
    "name":        "Mon Module",
    "version":     "1.0.0",
    "icon":        "icone.svg",
    "description": "Description courte affichée dans le dashboard.",

    "actions": [
        {
            "visual": "ma_table.xml",
            "functions": [
                {
                    "name":          "creerEntree",
                    "description":   "Créer une nouvelle entrée.",
                    "parameters":    [],
                    "logic":         [{ "type": "CREATE" }],
                    "actUponObject": "ma_table.xml"
                },
                {
                    "name":          "listerEntrees",
                    "description":   "Lister toutes les entrées.",
                    "parameters":    [],
                    "logic":         [{ "type": "LIST" }],
                    "actUponObject": "ma_table.xml"
                }
            ]
        }
    ],

    "page":        "mon_ui.json",
    "dataObjects": ["ma_table.xml"],
    "permissions": ["read:mon-module", "write:mon-module", "delete:mon-module"]
}

Champs racine

ChampRequisDescription
idouiIdentifiant unique — snake-case ou kebab-case, sans espaces
nameouiNom affiché dans le dashboard
versionnonSemver libre
iconnonFichier SVG ou image dans le dossier du module
descriptionnonTexte court affiché dans le dashboard
actionsouiTableau des groupes de fonctions
pageouiFichier JSON de l'interface
dataObjectsouiListe des fichiers XML de schémas utilisés
permissionsnonPermissions déclaratives (non encore enforced)

Paramètre d'une fonction (parameters[])

{
    "name":        "id",
    "type":        "EQUAL",
    "description": "ID de l'entrée",
    "optional":    false
}
TypeComportement
EQUALFiltre WHERE col = valeur en base de données
NONEPassé à l'action mais non utilisé comme filtre SQL
⚠️
actUponObject est obligatoire pour chaque fonction. Le backend échoue au démarrage si le fichier XML référencé n'existe pas.

2 Schéma XML — Base de données

Le backend génère automatiquement une table SQL à partir de ce fichier lors du démarrage.

<ma_table>

    <!-- Colonnes auto-générées (toujours inclure) -->
    <autoGenerated>
        <id/>
        <createdAt/>
        <updatedAt/>
    </autoGenerated>

    <!-- Colonne simple -->
    <col>
        <naming val="titre"/>
        <typing val="string"/>
    </col>

    <!-- Colonne avec contraintes -->
    <col>
        <naming   val="slug"/>
        <typing   val="string"/>
        <unique   val="true"/>
        <nullable val="false"/>
    </col>

    <!-- Colonne numérique -->
    <col>
        <naming val="score"/>
        <typing val="integer"/>
    </col>

</ma_table>

Types de colonnes

Valeur typingType SQL généré
stringVARCHAR / TEXT
integerINTEGER
booleanBOOLEAN
floatDOUBLE / FLOAT

Contraintes disponibles

BaliseValeurEffet
<unique>"true"Contrainte UNIQUE sur la colonne
<nullable>"false"Contrainte NOT NULL

Colonnes auto-générées

BaliseContenu généré
<id/>UUID auto-généré, clé primaire
<createdAt/>Timestamp de création
<updatedAt/>Timestamp de dernière mise à jour
<file/>Stockage binaire (pour modules avec upload de fichier)

3 UI JSON — Interface utilisateur

{
  "page": {
    "id":      "mon_module_page",
    "layout": "vertical",

    "bindings": {
      "maListe": "listerEntrees"    // nom local → nom de fonction
    },

    "state": {
      "champRecherche": ""           // variables de l'UI
    },

    "components": [...]
  }
}

bindings

Associe un nom local à une fonction du module. Ce nom est utilisé comme source.binding dans les composants. Le binding est rechargé automatiquement au montage de la page.

state

Valeurs initiales des variables de l'interface. Lisibles et modifiables via {{nomVariable}} dans les props et params.

4 Composants UI disponibles

HeaderTitre de page
ActionBarBarre d'actions
ButtonBouton cliquable
TextInputChamp de texte
ListListe d'éléments
IconButtonBouton icône
FileUploadZoneZone de dépôt
ModalFenêtre modale
ImageViewerAffichage image

Header

{ "type": "Header", "props": { "title": "Mon Module" } }

ActionBar

Barre horizontale qui contient d'autres composants (Button, TextInput…).

{ "type": "ActionBar", "components": [...] }

Button

{
  "type": "Button",
  "props": {
    "label":        "Rafraîchir",
    "icon":         "refresh",      // icône Lucide
    "uploadTarget": "mon-upload"   // déclenche une FileUploadZone
  },
  "action": { "action": "listerEntrees", "method": "POST" }
}

Icônes disponibles (Lucide) : "refresh", "upload", "trash", "plus", "fullscreen", "search"…

TextInput

Champ de texte avec bouton. Lié au state via stateKey. L'action se déclenche sur Enter ou clic du bouton.

{
  "type": "TextInput",
  "props": {
    "stateKey":    "champRecherche",   // obligatoire
    "placeholder": "Rechercher...",
    "label":       "Ville",
    "buttonLabel": "Go"
  },
  "action": {
    "action": "rechercherEntrees",
    "method": "POST",
    "params": { "query": "{{champRecherche}}" }
  }
}

List

Liste d'éléments depuis un binding. {{variable}} accède aux champs de chaque ligne. {{id}} est toujours disponible.

{
  "type": "List",
  "props": { "emptyMessage": "Aucun élément." },
  "source": { "binding": "maListe", "method": "POST" },
  "item": {
    "type": "ListItem",
    "props": { "title": "{{nom}}", "subtitle": "{{description}}" },
    "defaultClick": {
      "action": "lireEntree", "method": "POST",
      "params": { "id": "{{id}}" },
      "then": { "setState": { "entreeSelectionnee": "{{result}}" } }
    },
    "actions": [
      {
        "type": "IconButton",
        "props": { "icon": "trash", "tooltip": "Supprimer" },
        "action": [
          { "action": "supprimerEntree", "method": "POST", "params": { "id": "{{id}}" } },
          { "action": "listerEntrees",  "method": "POST" }
        ]
      }
    ]
  }
}

FileUploadZone

Zone de dépôt de fichier. Activée par un Button avec uploadTarget correspondant à l'id.

{
  "type": "FileUploadZone",
  "props": { "id": "mon-upload", "accept": "image/*", "multiple": false },
  "action": [
    { "action": "ajouterFichier", "method": "POST" },
    { "action": "listerEntrees",  "method": "POST" }
  ]
}

Modal

visible accepte une expression {{...}} évaluée sur le state.

{
  "type":    "Modal",
  "visible": "{{entreeSelectionnee != null}}",
  "content": {
    "type":   "ImageViewer",
    "source": { "binding": "getFichier", "method": "POST" },
    "params": { "id": "{{entreeSelectionneeId}}" }
  },
  "actions": [
    {
      "type": "Button",
      "props": { "label": "Fermer" },
      "action": { "type": "setState", "target": "entreeSelectionnee", "value": null }
    }
  ]
}

5 Types d'actions backend

Déclarés dans logic[].type de chaque fonction.

TypeComportement
LIST Retourne toutes les lignes de la table
READ Retourne une ligne filtrée par les paramètres EQUAL
CREATE Insère une nouvelle ligne avec les paramètres fournis
DELETE Supprime la/les lignes filtrées par les paramètres EQUAL
UPLOAD_FILE Upload un fichier binaire et crée une entrée en base
GET_FILE Retourne le fichier binaire d'une entrée
FETCH_EXTERNAL Appelle une API HTTP externe, parse la réponse JSON, upsert le résultat en base. Lit apiKey, baseUrl depuis params.json.

6 params.json — Paramètres configurables

Optionnel. Expose des valeurs configurables depuis l'UI via l'icône ⚙️ de la carte module dans le dashboard.

{
    "parameters": [
        {
            "key":          "apiKey",
            "label":        "Clé API",
            "type":         "secret",
            "defaultValue": "",
            "description":  "Clé secrète pour l'accès à l'API externe"
        },
        {
            "key":          "baseUrl",
            "label":        "URL de base",
            "type":         "string",
            "defaultValue": "https://api.exemple.com/v1",
            "description":  "Endpoint de l'API"
        }
    ]
}
TypeComportement
stringChamp texte standard, valeur visible
secretChamp texte masqué (password), valeur cachée dans l'UI

Lire un paramètre côté backend (FETCH_EXTERNAL)

ℹ️
Utile uniquement si vous créez un nouveau type d'action Kotlin. Pour les modules purement déclaratifs, ce n'est pas nécessaire.
val apiKey = globalParametersService.getParam(moduleId, "apiKey")
    ?: return mapOf("error" to "Paramètre 'apiKey' non configuré")

7 Actions chaînées

Passer un tableau à action exécute les actions en séquence. Pattern classique : modifier puis rafraîchir la liste.

"action": [
  { "action": "supprimerEntree", "method": "POST", "params": { "id": "{{id}}" } },
  { "action": "listerEntrees",  "method": "POST" }
]

Exemple complet — module "Tâches"

modules/tasks/
├── manifest.json
├── tasks.xml
└── tasks_ui.json

manifest.json

{
    "id": "tasks", "name": "Tâches", "version": "1.0.0",
    "description": "Gestionnaire de tâches simple.",
    "actions": [{
        "visual": "tasks.xml",
        "functions": [
            {
                "name": "createTask", "description": "Créer une tâche.",
                "parameters": [{ "name": "title", "type": "NONE" }, { "name": "done", "type": "NONE" }],
                "logic": [{ "type": "CREATE" }], "actUponObject": "tasks.xml"
            },
            {
                "name": "listTasks", "description": "Lister toutes les tâches.",
                "parameters": [], "logic": [{ "type": "LIST" }], "actUponObject": "tasks.xml"
            },
            {
                "name": "deleteTask", "description": "Supprimer une tâche.",
                "parameters": [{ "name": "id", "type": "EQUAL" }],
                "logic": [{ "type": "DELETE" }], "actUponObject": "tasks.xml"
            }
        ]
    }],
    "page": "tasks_ui.json",
    "dataObjects": ["tasks.xml"],
    "permissions": ["read:tasks", "write:tasks", "delete:tasks"]
}

tasks.xml

<tasks>
    <autoGenerated>
        <id/> <createdAt/> <updatedAt/>
    </autoGenerated>

    <col>
        <naming val="title"/>
        <typing val="string"/>
        <nullable val="false"/>
    </col>

    <col>
        <naming val="done"/>
        <typing val="boolean"/>
    </col>
</tasks>

tasks_ui.json

{
  "page": {
    "id": "tasks_page", "layout": "vertical",
    "bindings": { "taskList": "listTasks" },
    "state": { "newTaskTitle": "" },
    "components": [
      { "type": "Header", "props": { "title": "Tâches" } },
      {
        "type": "ActionBar",
        "components": [{
          "type": "TextInput",
          "props": { "placeholder": "Nouvelle tâche...", "stateKey": "newTaskTitle", "buttonLabel": "Ajouter" },
          "action": [
            { "action": "createTask", "method": "POST", "params": { "title": "{{newTaskTitle}}", "done": false } },
            { "action": "listTasks", "method": "POST" }
          ]
        }]
      },
      {
        "type": "List",
        "props": { "emptyMessage": "Aucune tâche pour l'instant." },
        "source": { "binding": "taskList", "method": "POST" },
        "item": {
          "type": "ListItem",
          "props": { "title": "{{title}}" },
          "actions": [{
            "type": "IconButton",
            "props": { "icon": "trash", "tooltip": "Supprimer" },
            "action": [
              { "action": "deleteTask", "method": "POST", "params": { "id": "{{id}}" } },
              { "action": "listTasks", "method": "POST" }
            ]
          }]
        }
      }
    ]
  }
}

Checklist de création

  • Dossier créé dans le répertoire des modules (HOMELAB_MODULES_SCAN_PATH)
  • manifest.json — champs id, name, actions, page, dataObjects renseignés
  • Chaque fonction a un actUponObject pointant vers un fichier XML existant
  • Fichier(s) XML créés avec au moins <autoGenerated><id/></autoGenerated>
  • Fichier UI JSON créé avec page.id et page.components
  • Backend redémarré pour scanner et charger le nouveau module
💡
Besoin d'appeler une API externe ? Utilisez le type d'action FETCH_EXTERNAL (voir module weather comme référence) et ajoutez un params.json pour stocker la clé API.
