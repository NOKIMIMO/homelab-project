# Functional Specification --- Homelab

## 1. Introduction

**Homelab** is a fully modular, self-hosted service orchestration platform. A **core** (Kotlin / Spring Boot) handles authentication, storage, database and telemetry; independent **modules** --- plain folders declared by a `manifest.json` --- plug into it without ever touching the core's code.

TL;DR: Homelab lets you create and expose new functionality (photo storage, a reader, a weather API call, etc.) by writing JSON/XML, not Kotlin.

---

## 2. Vision and goals

- **Zero code for a standard module**: a module is described with a `manifest.json` + an XML schema (+ an optional JSON UI page) --- no core recompilation.
- **Auto-generated database**: columns declared in the XML directly produce the SQL table, its constraints and its relations.
- **Centralized security**: JWT authentication handled by the core, sign-ups subject to admin approval.
- **Real extensibility via SDK**: for logic beyond the standard actions, `homelab-sdk` (Kotlin) exposes the required types (schema, relations, filters, actions, parameters).

---

## 3. Architecture

- **Core (`homelab-backend`)**: Kotlin 2.1 / Spring Boot --- module scanning, JWT auth, per-module SQL schema generation, system telemetry (OSHI), REST API.
- **Frontend (`homelab-frontend`)**: React 19 / TypeScript / Vite, DaisyUI + Tailwind (*night* theme), React Router v7.
- **SDK (`homelab-sdk`)**: Kotlin library shared between the core and a module's custom logic (not yet published to a public repository).
- **Showcase (`vitrine`)**: the project's presentation and documentation site, deployed separately (Cloudflare Workers).

---

## 4. Creating a module

Two ways to do it, covered in detail in `docs/MODULE_CREATION_GUIDE.md` and on the vitrine site:

1. **The module builder** --- an admin-only API (`/api/admin/module-builder`, `ModuleBuilderService`) that generates `manifest.json`, the XML schema and a UI page from a plain JSON description (tables, columns, relations, constraints).
2. **Manual format** --- hand-write `manifest.json` (id, name, actions, dataObjects, page, `uIFormat`, dependencies, permissions) and the XML schema (`<col>`, `<autoGenerated>`, `<linksTo>` for relations, `<constraints><uniqueTogether>` for composite constraints).

### Relation cardinality (`<linksTo>`)

`one-to-one` · `one-to-many` · `many-to-one` · `many-to-many`, with optional `cascadeDelete` / `cascadeUpdate` and cross-module relations via `moduleName`.

### Backend action types (`logic[].type`)

| Type | Behavior |
|---|---|
| `LIST` | Returns every row in the table |
| `READ` | Returns a row filtered by `EQUAL` parameters |
| `CREATE` | Inserts a new row |
| `DELETE` | Deletes the filtered row(s) |
| `UPLOAD_FILE` | Uploads a binary file, creates an entry |
| `GET_FILE` | Returns a row's binary file |
| `FETCH_EXTERNAL` | Calls an external HTTP API, upserts the result |

None of these standard types require Kotlin code.

### User interface (`uIFormat`)

- `JSON` --- page driven by a JSON file (bindings, state, ready-made components: Header, ActionBar, Button, List, Modal, ImageViewer…). **Still being stabilized (beta).**
- `STANDALONE` --- a dedicated frontend that calls the module's API directly (the core hands it an API key).
- `API` --- no UI served by the core (API-only module).

---

## 5. Authentication and administration

- Login by username/password issuing a JWT; sign-up subject to admin approval.
- Admin panel: live application logs, access management (approve/revoke accounts), global application settings.
- Each module can also expose its own configurable parameters (`params.json`: API keys, URLs, masked secrets), editable from the UI with no restart.

---

## 6. Telemetry

CPU, RAM, storage and uptime in real time via the **OSHI** library, reported by the core and shown on the dashboard.

---

## 7. Included modules (demo)

| Module | Role | Actions |
|---|---|---|
| **Photos** | Upload, storage and viewing of photos | `UPLOAD_FILE`, `GET_FILE`, `LIST`, `DELETE` |
| **Reader** | Photo reader organized into series and chapters, depends on the Photos module | `CREATE`, `READ`, `LIST` |
| **Weather** | Calls OpenWeatherMap, caches results in the database, configurable API key | `FETCH_EXTERNAL`, `LIST` |

---

## 8. Deployment

- **All-in-one Docker image** (frontend + backend + Postgres), published on GitHub Container Registry: `ghcr.io/nokimimo/homelab-project`. Automatically published (GitHub Actions) on every merge to `master`, with semantic versioning derived from commit messages.
- **Docker Compose** (`docker-compose.yml` / `docker-compose.dev.yml`) for a classic multi-container deployment or hot-reload development.

---

## 9. Who is this project for?

Self-hosting enthusiasts, developers and sysadmins who want to expose custom functionality without maintaining a full application per service.

### Actors

- **User**: accesses the modules they've been granted.
- **Admin**: creates/manages modules, approves sign-ups, reviews logs and telemetry.

---

## 10. Roadmap (not implemented yet)

- Third-party plugin marketplace with sandboxing (declared capabilities, FS/DB access arbitrated by the core).
- Mobile app (React Native or Flutter --- set aside until the rest of the project is mature enough).
- Additional modules under consideration: web SQL database explorer, video library, automated sync/download flows.
